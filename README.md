it's about Sales Analytics in Amazon India
# Amazon India: A Decade of Sales Analytics

End-to-end e-commerce analytics pipeline on ~1M Amazon India transactions
(2015-2025): data cleaning → EDA → SQL warehouse → PowerBI dashboard.

This README covers **Part 1 — Data Cleaning**, **Part 2 — EDA**, and
**Part 3 — SQL warehouse**. PowerBI dashboard is the last piece.

## Setup

```bash
# 1. Create a virtual environment
python -m venv .venv
.venv\Scripts\activate           # Windows
# source .venv/bin/activate      # macOS/Linux

# 2. Install dependencies
pip install -r requirements.txt

# 3. Drop your raw CSVs into data/raw/
#    (the products catalog + amazon_india_2015.csv ... amazon_india_2025.csv)

# 4. Run the cleaning pipeline
python src/run_cleaning.py
```

The script will:

1. Read all year-wise CSVs and concatenate them
2. Apply every cleaning function (Q1–Q10) in order
3. Join the cleaned transactions with the product catalog
4. Save `data/cleaned/amazon_india_cleaned.csv`
5. Write `reports/quality_report.txt` (before/after comparison)

## Cleaning questions covered

| # | What it fixes | Function | Source column |
|---|---|---|---|
| Q1 | Mixed date formats (DD/MM/YYYY, DD-MM-YY, ISO, invalid) → YYYY-MM-DD | `fix_dates` | `order_date` |
| Q2 | ₹ symbols, commas, "Price on Request" → numeric INR | `fix_prices` | `original_price_inr` |
| Q3 | "5.0", "4 stars", "3/5" → numeric 1.0–5.0 | `fix_ratings` | `customer_rating` |
| Q4 | "Bangalore"→"Bengaluru", "Bombay"→"Mumbai", spelling errors | `fix_cities` | `customer_city` |
| Q5 | Y/N, 1/0, True/False → consistent True/False | `fix_booleans` | `is_prime_member`, `is_prime_eligible`, `is_festival_sale` |
| Q6 | "Electronics/Electronic/ELECTRONICS" → canonical category | `fix_categories` | `category` |
| Q7 | "Same Day", "1-2 days", negatives, >30 days → numeric | `fix_delivery_days` | `delivery_days` |
| Q8 | Bulk-order vs duplicate-error detection (≤5 = bulk, >5 = error) | `handle_duplicates` | `(customer_id, product_id, date, amount)` |
| Q9 | Decimal-point errors (price 100× too high/low) | `fix_price_outliers` | `original_price_inr` |
| Q10 | "PhonePe/GooglePay" → "UPI", "CC/CREDIT_CARD" → "Credit Card" | `fix_payment_methods` | `payment_method` |

## How to use individual cleaners

If you only want to run one cleaning step (great for understanding each):

```python
import pandas as pd
from src.data_cleaning import fix_dates, fix_prices, fix_cities

df = pd.read_csv("data/raw/amazon_india_2020.csv")
df = fix_dates(df)
df = fix_prices(df)
df = fix_cities(df)
print(df[["order_date", "original_price_inr", "customer_city"]].head())
```

## Part 3 — SQL warehouse (MySQL)

A star-schema warehouse loaded from the cleaned CSV. PowerBI connects
straight to the dashboard views.

### Schema design

```
                    ┌──────────────┐
                    │   dim_time   │  one row per calendar day
                    └──────┬───────┘
                           │
   ┌──────────────┐    ┌──┴──────────────────┐    ┌──────────────┐
   │ dim_products ├────► fact_transactions   ◄────┤ dim_customers│
   └──────────────┘    └─────────────────────┘    └──────────────┘
```

| Table | Rows (~) | Key columns |
|---|---|---|
| `dim_time` | 4,000 | `date_id` (YYYYMMDD), year, quarter, month, week, day_of_week |
| `dim_products` | 2,000 | `product_id`, category, subcategory, brand, launch_year |
| `dim_customers` | ~50,000 | `customer_id`, customer_city, city_tier, age_group, first/last purchase |
| `fact_transactions` | ~1,000,000 | `transaction_id`, FKs, prices, payment, return, festival flags |

### 1. Set up MySQL credentials

Copy the template and fill in your password:

```powershell
copy .env.example .env
notepad .env
```

```
DB_HOST=localhost
DB_PORT=3306
DB_USER=root
DB_PASSWORD=your_actual_password
DB_NAME=amazon_india_db
```

### 2. Create the database + tables

```powershell
mysql -u root -p < sql\schema.sql
```

(In PowerShell use `Get-Content sql\schema.sql | mysql -u root -p` if `<` doesn't work.)

### 3. Load the cleaned data

```powershell
python src\run_db_load.py
```

This runs the full ETL: builds the 3 dimension tables (dedup'd), the
fact table (with FKs + date_id), and bulk-inserts everything in chunks.
Expected duration on a laptop: 5–10 minutes for 1M rows.
